The first problem wasn't design. It was figuring out which screen was the real one.

3BB Member already had an app — millions of users, established flows people used every day without thinking. I worked inside a shared design file holding years of flows, versions, and archived pages stacked on top of each other. Not every screen needed updating. Most didn't. But no one had clearly marked which was current.

01

Map current vs. legacy before touching anything

I separated active states from inherited content, then updated only where migration logic, service state, or implementation requirements actually changed the user's path.

— restraint is a design decision

02

Treat errors as first-class screens

Login errors, registration conditions, billing options — these aren't edge cases. Each state got clear copy, hierarchy, and a next action that QA and dev could implement consistently.

— edge cases are where trust is built

"Map the inherited-vs-contribution boundary on day one."
